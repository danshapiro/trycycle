This skill is invoked in trycycle Step 6 ("Plan with trycycle-planning"). A dedicated planning subagent is spawned and told to use this skill to produce an implementation plan.

**What the orchestrator already handles:**
- The orchestrator spawns a dedicated subagent for planning. The subagent is told not to spawn additional subagents.
- After the initial plan is written, the orchestrator may spawn fresh planning subagents for plan-editor rounds. Each fresh subagent receives the same task input plus the current plan path and must either declare the plan already excellent unchanged or improve it directly.
- The orchestrator wraps this skill with separate prompt templates for initial planning and plan-editor rounds. Keep those wrappers distinct; they share context, but their jobs are meaningfully different.
- In the plan-editor loop, each fresh planning subagent must either leave the plan already excellent unchanged or make it excellent itself. If it is not deemed already excellent, the loop continues with a fresh plan-editor cycle.

**What to strip:**

1. **Remove brainstorming preconditions.** The line "This should be run in a dedicated worktree (created by brainstorming skill)" should change to something like "This should be run in a dedicated worktree" (drop the brainstorming reference). The orchestrator creates the worktree in Step 4.

2. **Remove the plan header line about executing-plans.** The template header says `> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.` — change this to reference `trycycle-executing`.

3. **Keep the TDD emphasis, but adapt the test-step template.** The bite-sized task granularity, the plan document header format, and the "Remember" list are all core operational content the planning subagent needs. Keep the red/green TDD philosophy, exact file paths, and complete code in plan. But do not hard-code "write a new failing test" as the default first move for every task. The subskill should teach: use high-value existing automated checks when they already cover the relevant behavior; extend them when that is the clearest path; write new failing tests when coverage is missing. If the task or problem statement already names automated checks that are red and need to go green, those checks must appear explicitly in the plan.
